The Problem

LLMs call tools. But do they call the right tools?

Model Updates Break Things

A new Claude/GPT release might change tool selection, parameter passing, or response format. How do you catch regressions?

Cost Drift is Invisible

Switching from Sonnet to Haiku saves 80% on cost. But does accuracy drop? By how much? For which tools?

MCP Service Changes

When we update our MCP tools (add params, change schemas), do LLMs still call them correctly?

No Standard Framework

There's no pytest for MCP. Teams manually test with Claude Desktop and hope for the best.

Solution

TestMCPy: pytest for MCP

Write YAML tests. Run against any model. Get scores, costs, and regressions.

Deep Dive

Adaptive Testing: Prompt Mutations

When "List the first 5 datasets" fails, automatically explore why:

Original "List the first 5 datasets" FAIL
Paraphrase "Show me 5 datasets" PASS
Simplify "List datasets" PASS
Constrain "List the first 3 datasets" FAIL
Role "As a data analyst, list the first 5 datasets" PASS
Noise "List teh frist 5 datsets" PASS

Insight: The model fails when asked to pass a page_size parameter with words like "first N". The tool works fine — it's a prompt sensitivity issue, not a capability gap.

Why This Matters

Impact

  • Catch regressions before users do. Every MCP PR runs evals automatically. No more "Claude stopped listing charts after the update."
  • Optimize costs with data. Know exactly what switching from Sonnet to Haiku costs in accuracy. Make informed decisions, not guesses.
  • Understand prompt sensitivity. Mutation testing reveals which prompts are fragile and which are robust across models.
  • Open source leadership. First comprehensive MCP eval framework. Sets the standard for how MCP services should be tested.
